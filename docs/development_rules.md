
# Development Rules

이 문서는 프로젝트를 시작할 때 사람과 Codex가 함께 지킬 개발 약속을 정리하는 템플릿입니다.

## 기본 약속

- 기존 코드, 문서, 사용자 변경사항을 먼저 확인한다.
- 사용자가 만든 변경사항은 명시 요청 없이 되돌리지 않는다.
- 작업 범위와 관련 없는 리팩터링은 하지 않는다.
- 기능을 만들기 전에 현재 버전의 `plan_*.md`와 `checklist_*.md`를 확인한다.
- 큰 결정을 임시로 처리하지 않고 관련 문서에 남긴다.
- 검증하지 못한 항목은 완료한 것처럼 표시하지 않는다.

## Codex 작업 약속

- Codex는 작업 시작 시 `docs/README.md`를 먼저 읽는다.
- Codex는 현재 버전 폴더가 있으면 해당 버전의 계획, 체크리스트, 브랜치 로그를 확인한다.
- Codex는 파일 하나에 기능을 몰아넣지 않고 `file_structure_rules.md` 기준으로 분리한다.
- Codex는 기능별 작업 단위가 끝나면 커밋 후보 메시지와 검증 결과를 정리한다.
- Codex는 `develop`과 `main`에 직접 병합하지 않는다.
- Codex는 사용자가 PR을 만들 수 있도록 변경 범위, 체크리스트, PR 초안을 정리한다.

## Git 자동화 경계

Codex가 자동으로 제안하거나 수행할 수 있는 일:

- 현재 상태 확인
- 버전 폴더와 버전 문서 생성
- 작업 브랜치 이름 제안
- feature 브랜치 생성
- 기능별 커밋 메시지 제안
- 체크리스트 점검

사용자 승인 후에만 수행할 일:

- `feature/*`를 `version/*`에 병합
- `version/*` 또는 작업 브랜치를 원격에 push
- PR 생성

## 작업 완료 기준

- 관련 체크리스트가 갱신되어 있다.
- 변경 범위가 커밋 단위로 설명 가능하다.
- 실행, 빌드, 테스트 중 가능한 검증을 수행했다.
- 수행하지 못한 검증은 이유를 남겼다.
- 다음 버전으로 넘길 일이 있으면 버전 문서에 남겼다.
