# PR 작성 규칙

이 문서는 Pull Request를 만들 때 사용할 기준을 정리한다.

커밋은 실제 변경 단위를 기록하고, PR은 여러 커밋을 묶어 **무엇을 왜 바꿨는지 설명하는 작업 단위**로 사용한다.

## 1. PR 기본 원칙

- PR은 하나의 목적을 가진다.
- 서로 다른 목적의 작업은 가능하면 PR을 나눈다.
- 기능, 문서, 구조 변경이 섞이면 PR 설명에서 범위를 명확히 적는다.
- 아직 검토가 필요한 작업은 draft PR로 만든다.
- PR 제목만 보고 변경 목적을 이해할 수 있어야 한다.

## 2. 커밋과 PR 생성 운영 규칙

- Codex가 파일을 수정한 작업은 사용자가 따로 요청하지 않아도 작업 단위가 끝나면 바로 커밋한다.
- 커밋 메시지는 `docs/commit_convention.md`의 Gitmoji 규칙을 따른다.
- PR은 Codex가 직접 push하거나 생성하지 않는다.
- PR이 필요한 경우 Codex는 PR 제목과 본문 초안만 출력한다.
- 사용자가 직접 push하고 GitHub에서 PR을 생성한다.

이 규칙은 로컬 변경 사항을 깔끔하게 저장하면서도, 원격 저장소 반영과 PR 생성은 사용자가 직접 통제하기 위한 기준이다.

## 3. PR 제목 규칙

PR 제목은 아래 형식을 사용한다.

```txt
<깃모지> <작업 분류> : <작업 내용>
```

예시:

```txt
📝 문서 수정 : 프로젝트 운영 문서 정리
🎉 초기 설정 : React 프로젝트 생성
✨ 기능 추가 : 버튼 추가 흐름 구현
🧱 구조 변경 : builder feature 폴더 구조 분리
💄 UI 수정 : 캔버스 레이아웃 정리
```

작업 분류는 다음 값을 우선 사용한다.

| Gitmoji | 작업 분류 | 의미 |
| --- | --- | --- |
| 📝 | 문서 수정 | README, docs 문서 추가/수정 |
| 🎉 | 초기 설정 | 프로젝트 생성, 패키지 설정, 기본 구조 |
| ✨ | 기능 추가 | 사용자 기능 추가 |
| 🧱 | 구조 변경 | 폴더 구조, 파일 분리, 아키텍처 변경 |
| 💄 | UI 수정 | 화면 구성, 스타일, 사용성 개선 |
| 🐛 | 버그 수정 | 오류 수정 |
| ✅ | 테스트 | 테스트 추가/수정 |
| ♻️ | 리팩토링 | 동작 유지 구조 개선 |

PR 제목의 Gitmoji와 작업 분류는 `docs/commit_convention.md`와 같은 기준을 사용한다.

## 4. PR 본문 템플릿

PR 본문은 아래 형식을 기본으로 사용한다.

```md
## 변경 내용

- 

## 변경 이유

- 

## 확인한 것

- [ ] 

## 관련 문서

- 

## 다음 작업

- 
```

## 5. 항목별 작성 기준

### 변경 내용

이번 PR에서 실제로 바뀐 내용을 적는다.

예시:

- README를 프로젝트 index + savepoint 형식으로 정리했다.
- `docs/version_checklist.md`에 버전별 체크리스트를 추가했다.
- `features/builder` 중심의 파일 구조 기준을 문서화했다.

### 변경 이유

왜 이 변경이 필요한지 적는다.

예시:

- 구현 전에 프로젝트 방향과 파일 분리 기준을 고정하기 위해.
- `App.jsx`에 코드가 몰리는 것을 방지하기 위해.

### 확인한 것

직접 확인한 내용을 체크리스트로 적는다.

예시:

- [x] 문서 링크가 README에 연결되어 있다.
- [x] 버전 체크리스트와 파일 구조 규칙이 서로 충돌하지 않는다.
- [ ] `npm run build`는 아직 React 프로젝트 생성 전이라 실행하지 않았다.

### 관련 문서

이번 PR과 관련된 문서가 있으면 적는다.

예시:

- `docs/product_plan.md`
- `docs/file_structure_rules.md`
- `docs/version_checklist.md`

### 다음 작업

PR 이후 이어서 할 일을 적는다.

예시:

- React 프로젝트 초기 설정
- `v0.1.0` 파일 구조 생성

## 6. Draft PR 기준

다음 경우에는 draft PR로 만든다.

- 아직 구현이 끝나지 않은 경우
- 사용자의 확인이 필요한 경우
- 문서 기준은 잡았지만 세부 표현이 바뀔 수 있는 경우
- 빌드나 실행 확인이 아직 안 된 경우

다음 경우에는 ready PR로 바꿀 수 있다.

- 목표한 체크리스트가 끝난 경우
- 주요 확인 항목을 실행한 경우
- 다음 사람이 봐도 변경 범위를 이해할 수 있는 경우

## 7. PR을 나누는 기준

다음 경우에는 PR을 나누는 것을 우선 검토한다.

- 문서 변경과 기능 구현이 크게 섞이는 경우
- UI 변경과 데이터 구조 변경이 동시에 큰 경우
- 설정 변경과 기능 구현이 동시에 큰 경우
- 한 PR 설명이 너무 길어져서 목적이 흐려지는 경우

하지만 초기 문서 세트처럼 하나의 목적이 분명하면 한 PR로 묶어도 된다.

## 8. 커밋과 PR의 차이

커밋:

- 실제 변경 단위
- 가능한 작고 명확하게 작성
- Gitmoji 규칙을 따른다.

PR:

- 여러 커밋을 묶은 작업 단위
- 무엇을 왜 바꿨는지 설명
- 확인한 것과 다음 작업을 함께 기록

## 9. PR 초안 출력 전 확인

PR 초안을 출력하기 전 다음을 확인한다.

- [ ] 현재 브랜치가 작업 브랜치인지 확인했다.
- [ ] 관련 변경만 커밋에 포함했다.
- [ ] README 또는 docs 링크가 깨지지 않는다.
- [ ] 필요한 확인 명령을 실행했거나, 실행하지 못한 이유를 적었다.
- [ ] PR 제목과 본문이 변경 범위를 설명한다.
